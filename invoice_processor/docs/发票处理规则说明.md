# 发票智能筛选系统 - 规则说明文档

## 一、业务背景

本系统用于从个人发票池中智能筛选符合月度福利补贴要求的发票，支持餐饮、交通、通讯三种类型发票的自动化处理。

**核心特性**：
- 支持增量处理：可多次添加新发票，系统自动合并结果
- 智能选择算法：时间优先 + 小面额优先 + 智能跳过
- 自动校验规则：日期范围、抬头格式、黑名单等
- Markdown报告输出

## 二、补贴金额上限

| 发票类型 | 月度补贴上限 |
|----------|--------------|
| 餐饮     | 680元        |
| 交通     | 500元        |
| 通讯     | 300元        |

## 三、发票校验规则

### 3.1 时间规则

**规则**：发票开票日期应在报销月份前3个月内

**计算方式**：
- 报销月份：`YYYY-MM`
- 有效期：报销月份往前推2个月的第一天，到报销月份的最后一天
- 示例：报销2026年1月，有效期为2025年11月1日 ~ 2026年1月31日

**代码实现**：
```python
earliest_date = reimbursement_date - relativedelta(months=2).replace(day=1)
latest_date = reimbursement_date + relativedelta(months=0, day=31)
```

## 四、智能选择策略

### 4.1 选择目标

为每种发票类型选择发票，使得：
- 选中发票总额 **≥ 补贴上限**
- 超额部分 **尽可能小**
- 优先使用 **时间较早** 的发票
- 同期发票优先使用 **金额较小** 的

### 4.2 排序规则

**主排序**：按发票日期升序（早的在前）

**次排序**：同一日期的发票，按金额升序（小的在前）

```python
valid.sort(key=lambda x: (x.invoice_date, x.amount))
```

### 4.3 选择算法

#### 4.3.1 基础逻辑

```
对每种发票类型分别处理：
1. 筛选校验通过的发票
2. 按(日期, 金额)排序
3. 依次选择，直到达到或超过上限
4. 达到上限后，后续发票标记为"多余"
```

#### 4.3.2 智能跳过规则

**触发条件**：添加某张发票后会大幅超额

**判断标准**：
```
超额金额 = 当前累计 + 发票金额 - 补贴上限
如果 超额金额 > 补贴上限 × 50%：
    跳过这张发票，继续检查后续
否则：
    选择这张发票
```

**示例**：
- 补贴上限：680元
- 当前累计：575元
- 遇到450.50元发票：575 + 450.50 - 680 = 345.50 > 340 → **跳过**
- 遇到130元发票：575 + 130 - 680 = 25 < 340 → **选择**

### 4.4 选择示例

#### 场景1：同日期多张发票

| 日期 | 金额 | 决策 | 累计 |
|------|------|------|------|
| 1月10日 | 66元 | 选择 | 66元 |
| 1月11日 | 110元 | 选择 | 176元 |
| 1月11日 | 399元 | 选择 | 575元 |
| 1月11日 | 450.50元 | 跳过（超额345.50 > 340） | - |
| 1月24日 | 130元 | 选择 | 705元 ✓ |

**结果**：选中 66 + 110 + 399 + 130 = 705元，多余 450.50元

#### 场景2：不同日期发票

| 日期 | 金额 | 决策 | 说明 |
|------|------|------|------|
| 2025年11月 | 100元 | 优先选择 | 最早日期 |
| 2025年12月 | 200元 | 次优先 | 次早日期 |
| 2026年1月 | 300元 | 最后选择 | 最晚日期 |

## 五、增量处理机制

### 5.1 使用场景

当发票不足时，用户可以多次添加发票到invoice池，系统会自动合并结果：

```
第1次运行：
- invoice池: 10张发票
- 处理结果: 选中7张，多余3张
- 选中发票移入: output/2026-01/
- 多余发票保留: invoice/ (重命名为 [多余]xxx.pdf)

第2次运行（添加新发票）：
- invoice池: 新发票 + [多余]xxx.pdf
- 系统读取: 已选中7张（从 output/2026-01/）
- 处理: 新发票 + 重新评估[多余]发票
- 合并结果: 已选中 + 新选中
- 生成完整报告
```

### 5.2 文件状态流转

```
┌─────────────┐
│  invoice/   │ 原始发票池
│  xxx.pdf    │
└──────┬──────┘
       │ 处理
       ▼
┌─────────────────────────────────────┐
│  校验结果                            │
├──────────────┬──────────────┬────────┤
│ 选中         │ 多余         │ 错误   │
└──────┬───────┴──────┬───────┴───┬────┘
       │              │           │
       ▼              ▼           ▼
┌─────────────┐ ┌──────────┐ ┌──────────┐
│output/      │ │invoice/  │ │output/   │
│{YYYY-MM}/   │ │          │ │errors/   │
│类型_金额.pdf│ │[多余].pdf│ │[错误-xx] │
└─────────────┘ └──────────┘ └──────────┘

二次运行时：
- output/{YYYY-MM}/ 中的发票被读取为"已选中"
- invoice/ 中的发票（包括[多余]）被重新处理
- 结果合并后生成报告
```

### 5.3 多余发票的重新评估

为什么需要重新评估[多余]发票？

| 场景 | 说明 |
|------|------|
| **过期** | 之前的[多余]发票可能已超过3个月账期，变为错误 |
| **更合适** | 之前多余的发票，现在可能变得合适了 |
| **补充** | 添加新发票后，可能需要调整选择 |

**示例**：
```
第1次运行（1月5日）：
- 有效期: 2025-10-01 ~ 2026-01-31
- 选中: 10月、11月、12月的发票
- 多余: 1月的大额发票

第2次运行（1月25日）：
- 有效期: 2025-10-01 ~ 2026-01-31（不变）
- 10月的发票可能已过期（超过3个月）
- 需要重新选择
```

## 六、文件处理规则

### 6.1 文件状态分类

| 状态 | 说明 | 处理方式 |
|------|------|----------|
| **选中** | 通过校验且被选中用于当月报销 | 移动到 `output/{YYYY-MM}/` |
| **多余** | 通过校验但未选中（本次或之前） | 保留在 `invoice/`，重命名为 `[多余]...` |
| **错误** | 校验失败 | 移动到 `output/errors/`，重命名为 `[错误-原因]...` |

### 6.2 文件命名规则

#### 6.2.1 选中发票

```
{类型}_{金额}元_{销方名称}.pdf
```

示例：`餐饮_110.00元_武汉市渔焰餐饮有限公司.pdf`

#### 6.2.2 多余发票

```
[多余]{类型}_{金额}元_{销方名称}.pdf
```

示例：`[多余]餐饮_450.50元_武汉市武昌区弈柏长源火锅餐饮店.pdf`

**注意**：带`[多余]`前缀的发票会在下次运行时被重新处理和评估

#### 6.2.3 错误发票

```
[错误-{原因}]{类型}_{金额}元_{销方名称}.pdf
```

错误原因简码：
- `超期` - 超出3个月账期
- `抬头错误` - 公司名称/税号不符
- `黑名单` - 销方在黑名单中
- `未知类型` - 无法识别类型
- `识别失败` - OCR无法解析

示例：`[错误-超期]餐饮_399.00元_某某公司.pdf`

#### 6.2.4 文件名清理

所有销方名称都会经过清理：
- 移除换行符、制表符
- 移除Windows非法字符 `< > : " / \ | ? *`
- 限制长度为50字符
- 空名称替换为"未知"

### 6.3 目录结构

```
invoice_processor/
├── invoice/                 # 发票池（输入）
│   ├── *.pdf               # 原始发票
│   └── [多余]...pdf        # 多余发票（保留，下次重新评估）
└── output/
```

## 七、发票识别规则

### 7.1 PDF文本提取

使用 `pdfplumber` 库提取PDF中的文本内容。

### 7.2 关键信息提取

#### 7.2.1 购销方信息

**挑战**：PDF中购销方信息可能分行显示

**格式示例**：
```
购 名称：深圳前海微众银行股份有限公司 销 名称：武汉市渔焰餐饮有限公司
买 售
方 方
```

**提取策略**（按优先级）：
1. 尝试匹配标准格式：`购买方名称：xxx`
2. 匹配分行格式：`购\n名称：xxx`
3. 匹配"名称："后的公司名称（以"公司"、"有限"等结尾）
4. 提取税号：`统一社会信用代码/纳税人识别号：税号`

#### 7.2.2 项目名称

**格式**：`*类别*项目名称`

**示例**：
- `*餐饮服务*餐饮服务`
- `*电信服务*通信服务费`
- `*汽油*92号车用汽油`

**提取策略**：
1. 匹配 `*xxx*yyy` 格式
2. 查找"项目名称"标题后的数据行

#### 7.2.3 金额提取

- 价税合计（大写/小写）
- 优先使用不含税金额
- 如无单独金额，使用价税合计

### 7.3 类型识别

根据 **销方名称** + **项目名称** 进行匹配：

| 类型 | 销方/项目关键词 |
|------|----------------|
| 餐饮 | 餐饮、食品、餐饮服务 |

## 十二、常见问题

### Q1: 为什么有些发票被识别为"未知类型"？

**A**: 可能的原因：
- 发票项目名称不包含任何关键词
- OCR识别失败，无法提取项目名称
- 发票类型确实不在支持范围内

**解决**: 在 `INVOICE_TYPES` 中添加对应关键词

### Q2: 为什么达到上限后还在选择发票？

**A**: 系统会尽量达到或超过上限，但会智能跳过导致大幅超额的发票（超过上限50%）。这是为了避免浪费大额发票。

### Q3: 发票抬头检查失败怎么办？

**A**: 检查：
1. PDF中是否正确提取了购方信息
2. 公司名称和税号是否完全匹配（包括全称）
3. 是否是通讯发票（通讯发票允许个人抬头）

### Q4: [多余]发票会一直保留吗？

**A**: 是的，但每次运行都会重新评估：
- 如果已过期 → 标记为错误，移入 errors/
- 如果现在更合适 → 选中，移入 output/{月}/
- 如果仍然多余 → 继续保留在 invoice/

### Q5: 如何清空重新开始？

**A**: 删除或移动以下内容：
```bash
# 清空已选中发票
rm -rf output/2026-01/

# 清空多余发票（可选）
rm invoice/[多余]*.pdf

# 重新运行
python main.py --month 2026-01
```

### Q6: 报告中的"已选中"发票是累计的吗？

**A**: 是的。报告包含：
- 本次运行选中的发票
- 之前运行已选中的发票（从 output/{月}/ 读取）
- 两者的合计结果

### Q7: 为什么通讯发票显示个人抬头是警告？

**A**: 通讯发票规则特殊，允许个人抬头，但系统仍会显示警告提醒用户注意。

### Q8: 多余发票会过期吗？

**A**: 会。每次运行时，系统会重新检查所有发票（包括[多余]发票）的日期：
- 如果超过3个月账期 → 自动标记为错误
- 所以定期运行可以自动清理过期发票
